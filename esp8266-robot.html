WiFi-Teleoperated Robotic Chassis

SYSTEM ARCHITECTURE & BUILD LOG // ESP8266 + C++

This document details the engineering constraints, power distribution, and software architecture required to build a reliable, low-latency teleoperated vehicle using the ESP8266 microcontroller.

1. System Architecture & Control Loop

The core requirement was minimizing input latency over a local network. Standard HTTP requests require establishing and tearing down a connection for every command, which causes motor stuttering. To solve this:

  • Asynchronous Web Server: Hosted directly on the ESP8266, allowing non-blocking network requests while the main C++ loop continues to process motor PWM signals.
  • Data Pipeline: Mobile Browser (Client UI) → WiFi (802.11) → ESP8266 (Host) → 3.3V GPIO Logic → L298N Motor Driver.

2. Power Distribution & Isolation

A major failure point in embedded robotics is microcontroller brownouts caused by motor stall currents pulling the battery voltage below the logic threshold.

  • Motor Rail: Direct 7.4V feed from series 18650 Li-ion cells to the L298N driver to handle raw current spikes during sudden chassis acceleration.
  • Logic Rail (3.3V): The ESP8266 requires a strict, noise-free 3.3V. I stepped down the 7.4V source using a dedicated buck converter (or linear regulator), adding decoupling capacitors to filter out electrical noise from the brushed DC motors.

3. Hardware Debugging & Logic Challenges

The Challenge: Initial testing resulted in the ESP8266 resetting its WiFi connection every time the vehicle reversed direction.

The Fix: Diagnosed the issue as a sudden voltage drop caused by the reverse EMF (electromotive force) of the motors. By ensuring proper logic grounding and adding a flyback diode/capacitor filter to the logic rail, the MCU achieved stable continuous uptime.
